# Figures: *Agentic code review uncovers severe, widespread reproducibility issues at NeurIPS 2025*

Publication figures for the TMLR submission. Every figure is **anonymised** (no
paper named or ranked; worked examples are de-identified "Paper A/B/…") and leads
with **objective, re-runnable evidence** rather than the auditor's subjective
`severity` label.

## Build

Data computation is separated from rendering, so every number a figure shows is
auditable in one place (`figure_data.json`):

```bash
python _summary/aggregate.py             # → _summary/data/figure_data.json
python _summary/build_coverage_figure.py # (+ the other _summary/build_*_figure.py scripts) → figures/*.{png,pdf,svg}
```

Each figure is written as PNG (300 dpi), PDF (vector, for LaTeX `\includegraphics`)
and SVG (editable text). The scripts read only `audits/<paper>/findings.json` and
`findings_verified.json` (excluding `audits/theory/` and
`audits/excluded_justified_no_code/`) plus the cloned repos.

## What the figures rest on

Severity (high/med/low) is an LLM judgment, so it is **not** an axis of any figure.
Every figure rests on one of two objective axes:

1. **Deterministic pipeline state**, does the released code clear each reproduction
   gate? (filesystem + structured-field checks, binary)
2. **Evidence strength**, how each finding can be independently re-checked
   (quote+SHA / code location / re-runnable script / survived adversarial re-verification)

## Reproduction gates

Roots at *source present* and splits into two independent axes (neither gates the
other). Every gate is a filesystem check or a structured-`topic` match, no code is
executed and no severity label is read. Numbers are a live snapshot (currently 100
papers), regenerated by `aggregate.py`.

**Root (availability), of all 100 papers**

| Gate | Passes iff |
|---|---|
| Code URL linked | `code_links.txt` non-empty → **100** |
| Source code present | ≥1 cloned repo confirmed as the paper's own core (not a baseline/dependency) → **87** |

**Track A: environment readiness** (independent attributes of the 87 with source)

| Attribute | Passes iff | Count |
|---|---|---|
| Has runnable entrypoint | a `.py` named `train\|main\|run\|eval\|test\|demo\|experiment`, or any `.sh` | **78 / 87** |
| Ships install / environment spec | a dependency file (`requirements.txt` / `setup.py` / `Dockerfile` / …) or README-documented install | **66 / 87** |

**Track B: code coverage** (nested, of the 87 with source)

| Gate | Passes iff (nested) | Count |
|---|---|---|
| Source code present | (baseline) | **87 / 87** |
| Most of the code committed | no surviving `missing` finding signalling wholesale absence | **58 / 87** |
| All published results trace to committed code | additionally, no surviving `missing` "result traceability" finding | **6 / 87** |

A *surviving* finding is one not rejected by the verifier, not reclassified by the
supplement-fetch pass, and not dropped by the repo-provenance pass.

## How verification works

`start_verify_prompt.md` + `auditowl_verifier_prompt.md` define a two-stage
re-check that writes `audits/<paper>/findings_verified.json`:

1. **Sonnet first pass**, one fresh agent per paper re-verifies *every* finding,
   assuming it false until the evidence at the cited `file:line` proves it, running
   any cited `check_script`. Verdict ∈ {keep, lowered, reject}.
2. **Opus escalation**, for papers with any non-keep verdict, an Opus 4.8 agent
   re-judges only those findings.

Across the study: **603 kept · 2 kept with severity lowered · 1 rejected**.

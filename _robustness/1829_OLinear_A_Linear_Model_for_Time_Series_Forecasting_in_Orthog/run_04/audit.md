# Code-repository audit — OLinear (paper 1829)

## 1. Summary

The repository `code/jackyue1994__OLinear/` is the official implementation of
*OLinear: A Linear Model for Time Series Forecasting in Orthogonally Transformed
Domain*. It is a Time-Series-Library/iTransformer-derived codebase: a single
`run.py` driver, `experiments/exp_forecast.py` train/test loop,
`data_provider/data_loader.py` split/scaling logic, `model/OLinear.py` and
`model/OLinear_C.py` (the two headline models), several ablation models, and
~120 per-dataset shell scripts under `scripts/`. The core method (OrthoTrans)
relies on pre-computed orthogonal matrices `Qi`, `Qo` (temporal) and a channel
correlation matrix (`CorrMatv`), generated by `dataset/Generate_corrmat.ipynb`
and shipped as `.npy` files for the 11 datasets bundled in `dataset/`.

What I did:
- Read the paper's method (§4.2 OrthoTrans, §4.3 NormLin), implementation
  details (App. D), dataset-split table (Table 11), and robustness/ablation
  appendices.
- Read `run.py`, `exp_forecast.py`, `data_loader.py`, `data_factory.py`,
  `metrics.py`, `model/OLinear.py`, `model/OLinear_C.py`, and the corrmat
  notebook.
- Wrote `_audit_code/check_corrmat_train_only.py` to verify deterministically
  that the temporal Q matrix is computed only from training-set indices
  (no train/test leakage in the basis construction).

Overall the repo is reasonably complete and methodologically sound. The key
non-leakage claim holds: the orthogonal basis and the channel correlation matrix
are computed from the training portion only, the scaler is fit on training only,
the test loader does not drop the last batch, and early stopping/checkpoint
selection use validation loss (not test loss). Findings are minor: an
undocumented learnable bias in the transform, unpinned dependencies, several
headline-benchmark datasets/Q-files not bundled (download pointers provided), and
two latent test-set-selection code paths that are inert for the released models
but would activate test-metric selection for models returning intermediate
stages.

Note (why downgraded to `question`): for the two headline models the selection
is inert. `model/OLinear.py:152` and `model/OLinear_C.py:162` both `return out`
(a single tensor, not a tuple), so the `isinstance(outputs_list, tuple)`
guard at `exp_forecast.py:799` is False and `mse_list` receives only the final
metric (`exp_forecast.py:849`), making `min_index = 0`. Likewise
`run.py:463-487` initialises `test_batch_size_list = [args.test_batch_size]` and
only appends `args.test_batch_size` if absent (it is already present), so the
list always has exactly one element (all shipped scripts pass
`--test_batch_size 16`). I could not construct an input where these paths
select a non-final value for the released OLinear/OLinear-C configs, hence
`condition_satisfiable: false` and `status: question`.

## 5. Closing lists

**Top take-aways** (ranked):
1. (`difference`) `orthotrans-undocumented-delta` — the implemented transform adds learnable `delta1/delta2` biases absent from the paper's OrthoTrans equation.
2. (`missing`) `datasets-not-bundled` — ETT/ECL/Traffic/Solar/PEMS/Exchange and their Q-files are not in the repo; reproducing those headline rows needs external download + notebook regeneration.
3. (`methodology`) `test-set-selection-latent-paths` — test-driven argmin over decoder stages and test batch sizes exists but is inert for the released models (question).
4. (`missing`) `unpinned-dependencies` — `requirements.txt` has no version pins.
5. (`difference`) `train-epochs-patience-mismatch` — shipped scripts use 30 epochs / patience 5 vs paper's 50 / 10.

**Items that genuinely look fine** (actively checked):
- No leakage in the orthogonal basis: temporal Q (`Generate_corrmat.ipynb` cell 0) and channel `CorrMatv` (cell 1) are computed from the training rows only; verified deterministically (`_audit_code/check_corrmat_train_only.py`, indices `[0, num_train)` match `data_loader.py:268-272`).
- StandardScaler fit on training rows only for the main forecasting path (`data_loader.py:308,69-70,166`).
- Test loader uses `drop_last=False` (`data_factory.py:44`), so the test metric covers the full test set (avoids the common TSLib partial-batch bias).
- Early stopping / checkpoint selection use validation loss, not test loss (`exp_forecast.py:542-547`); test loss is only printed.
- Seeding sets random, torch, and numpy (`run.py:467-471`).
- MSE/MAE definitions are standard (`utils/metrics.py:14-19`); evaluation scale matches the iTransformer baseline convention.
- Dataset split ratios in code (6:2:2 ETT, 7:1:2 weather/ECL) match Table 11; Q-file ratios (0.6 ETT, 0.7 others) are consistent.
- Weighted-L1 training loss is disclosed in the paper ("following CARD", App. D) and matches `exp_forecast.py:156-191`.
- Ablation scripts/models for basis, var_temp, and attn_var are present.

**Open questions for the authors:**
- Are `delta1/delta2` ablated, and what is their effect on the reported numbers?
- Please confirm the test-batch-size / decoder-stage argmin (`exp_forecast.py:852-858`, `run.py:566-576`) never selected a non-final value for the reported OLinear/OLinear-C numbers.
- Which exact training schedule (epochs/patience) produced the headline tables, given the script vs App. D discrepancy?
